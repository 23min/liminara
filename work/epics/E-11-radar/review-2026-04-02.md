# E-11 Radar Review — 2026-04-02

## Summary

Verdict: request changes.

The current worktree improves a few real issues in M-RAD-03 and the Phase 5a Radar correctness slice, especially around rank determinism, `mix radar.run` usability, and DAG rendering polish. The main review conclusion is that the core replay contract is still not restored. The two largest gaps are:

1. replay for multi-decision recordable ops is still single-decision-per-node
2. replay still rebuilds a fresh Radar plan instead of reusing the recorded plan

That means the current implementation is not yet at a point where E-11c can be considered complete, and M-RAD-03 still cannot claim exact replay for generated briefings.

## Detailed Findings

### 1. Critical — Multi-decision replay is still broken

The review's highest-severity issue is unchanged from the known Phase 5a gap: replay for recordable ops still assumes one decision record per node and reconstructs one output from `output.response`.

Evidence:

- `runtime/apps/liminara_core/lib/liminara/decision/store.ex:55-63`
  - `Decision.Store.put/3` writes one file per `node_id`
- `runtime/apps/liminara_core/lib/liminara/decision/store.ex:130-132`
  - file layout is `decisions/<node_id>.json`
- `runtime/apps/liminara_core/lib/liminara/run.ex:249-266`
  - replay loads a single decision for a node and reconstructs a single `%{"result" => output_value}` artifact from `decision["output"]["response"]`
- `runtime/python/src/ops/radar_summarize.py`
  - summarize emits a list of decisions for a single node execution and also returns multi-key outputs such as `summaries` and `decisions`

Impact:

- the summarize replay contract remains broken
- exact replay of a full Radar briefing is still not possible
- E-11c's core acceptance criterion is still open

This matches the existing documented gap in:

- `work/gaps.md`
- `work/roadmap.md`
- `work/milestones/tracking/M-RAD-03-tracking.md`

### 2. High — Replay still rebuilds a fresh Radar plan

The runtime now persists `plan.json`, but the public replay entrypoint still calls `pack_module.plan(input)` and builds a new plan rather than loading the recorded one.

Evidence:

- `runtime/apps/liminara_core/lib/liminara.ex:59-77`
  - `Liminara.replay/4` rebuilds the plan from the pack input
- `runtime/apps/liminara_core/lib/liminara/run.ex:73`
  - direct execution now persists the original plan
- `runtime/apps/liminara_radar/lib/liminara/radar.ex:139-147`
  - rank gets a fresh `reference_time`
- `runtime/apps/liminara_radar/lib/liminara/radar.ex:157-166`
  - compose gets a fresh synthetic briefing id and date

Impact:

- replay can still drift even if decision storage is fixed
- pure-node inputs are not necessarily identical between original run and replay
- the milestone requirement that replay produce the same briefing remains unmet

This is a cross-file contract problem, not an isolated bug in any single op.

### 3. Medium — `run_id` provenance is still incorrect

The current changes improve the visible placeholder in `compose_briefing`, but they do not propagate the actual runtime `run_id` through the plan. The review does not consider this fixed.

Evidence:

- `runtime/apps/liminara_radar/lib/liminara/radar.ex:112-117`
  - dedup still receives `"run_id" => {:literal, "placeholder"}`
- `runtime/python/src/ops/radar_dedup.py:40-45`
  - dedup reads `run_id` from inputs
- `runtime/python/src/ops/radar_dedup.py:91-106`
  - dedup writes that `run_id` into persisted LanceDB rows
- `runtime/apps/liminara_radar/lib/liminara/radar.ex:159-166`
  - compose now receives a synthetic `briefing-YYYYMMDDTHHMMSS` id, not the actual runtime run id

Impact:

- persisted Radar history still contains incorrect run provenance
- briefing metadata is still not the runtime's real `run_id`
- traceability remains weaker than the milestone spec requires

This also means the tracking note:

- `work/milestones/tracking/M-RAD-03-tracking.md`

currently overstates the fix by marking the `run_id` shortcut as resolved.

### 4. Medium — Rank is only conditionally pure

The main Radar path now passes `reference_time`, which is an improvement. But the op still falls back to wall-clock time when `reference_time` is missing or invalid.

Evidence:

- `runtime/python/src/ops/radar_rank.py:81-87`
  - `_parse_reference_time` returns `datetime.now(timezone.utc)` for missing or malformed input

Impact:

- the op still has a latent ambient-time fallback
- the code is safer than before, but not strictly wall-clock-free in all cases
- the status text in the tracking doc is stronger than the implementation guarantee

This is not as severe as the replay issues because the current pack path supplies the value, but it is still worth recording as an unresolved edge in the determinism claim.

### 5. Low — The LLM dedup live-run fix is missing a regression test

`radar_llm_dedup.py` now handles either a full dedup result dict or a direct list of items.

Evidence:

- `runtime/python/src/ops/radar_llm_dedup.py:27-33`

This change looks reasonable, but the existing tests in:

- `runtime/python/tests/test_radar_llm_dedup.py`

only exercise the list-shaped input path.

Impact:

- a future cleanup could easily break the dict-shaped wiring again
- the live-run bug that motivated this change is not locked down by tests

## Acceptance Criteria Impact

### M-RAD-03

#### AC2: Rank

Status: improved, still not fully clean.

- The deterministic input path is now explicit.
- `historical_centroid` remains intentionally inert.
- source diversity remains cluster-level rather than per-item.
- invalid or missing `reference_time` still falls back to wall clock.

#### AC3: Summarize

Status: forward execution is fine, replay contract still fails.

- the summarize op itself can produce the expected outputs
- runtime replay still cannot faithfully reconstruct multi-decision, multi-output summarize behavior

#### AC4: ComposeBriefing

Status: partially met.

- structure assembly looks fine
- actual run metadata is still not the real runtime run id

#### Replay requirement

Status: not met.

The milestone spec explicitly says replay should produce identical briefing output. The current worktree still does not deliver that.

### E-11c Phase 5a Correctness

Status: not complete.

Remaining blockers from the review:

- multi-decision replay still broken
- replay still rebuilds a fresh time-stamped plan
- `run_id` tracking remains incorrect in actual data flow

## Non-Blocking Notes

- The generated `.claude/*` output and `.ai/sync.sh` changes look coherent as framework-sync work. I did not find a correctness problem there during this review.
- The web changes in `RunsLive.Show` and `root.html.heex` appear reasonable for readability and labeling. I did not find a blocking bug in those files from the current diff alone.
- The copied `dag-map-bundle.js` change set is large because it includes more of the upstream library surface, but the `dag-map` source-side tests passed and I did not find a review blocker there.

## Recommended Resolution Order

1. Fix decision fan-out and replay reconstruction for multi-decision recordable ops.
2. Change replay to use the stored plan rather than rebuilding a new Radar plan.
3. Thread the actual runtime `run_id` through plan execution, dedup persistence, and briefing metadata.
4. Tighten the rank op so missing or invalid `reference_time` does not silently fall back to wall clock.
5. Add a regression test for the dict-shaped `radar_llm_dedup.py` input.

## Final Assessment

The current worktree is useful progress, but it should not be treated as closing the replay and provenance correctness slice yet. The highest-risk issues are architectural contract problems rather than cosmetic defects, and they directly affect the core value proposition of Liminara: exact replay of recorded nondeterministic decisions.
